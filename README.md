# YClients Push Automation

Серверная автоматизация рассылки push-уведомлений клиентам через веб-кабинет
**YClients**. Берёт лиды из Google-таблицы → находит/создаёт клиента в базе YClients
через браузер (Playwright) → отправляет push → пишет результат обратно в таблицу.

Push-рассылки в YClients доступны только через UI (доходят клиентам с установленным
приложением и включёнными уведомлениями), поэтому отправка идёт строго через браузер,
а не через API.

---

## Как это работает

```
Google Sheets (строки: имя, телефон, текст push, статус)
  → run.py: polling каждые POLL_INTERVAL секунд, берёт строки со статусом «новый»
  → Playwright worker (persistent Chromium-профиль, залогиненная сессия YClients)
  → UI YClients: поиск клиента по телефону → создать при отсутствии (без дублей)
                 → проставить согласия → отправить PUSH в YPLACES
  → запись статуса обратно в таблицу
```

Строки обрабатываются **строго по одной** (конкуренция = 1) — это исключает дубли
клиентов и конфликты в UI.

### Статусы в колонке E

| Статус | Значение |
|---|---|
| `новый` | строка взята в работу (источник задач) |
| `в работе` | отправка началась; защищает от повторной обработки при сбое |
| `отправлено` | push отправлен успешно |
| `нет push-канала` | получателей 0 (слать некому) |
| `ошибка: <текст>` | сбой; текст ошибки рядом |

## Структура проекта

| Файл | Назначение |
|---|---|
| `run.py` | боевой воркер: polling таблицы + обработка строк (entrypoint в Docker) |
| `simple_run.py` | вся браузерная логика YClients + один ручной прогон для отладки |
| `sheets.py` | чтение/запись Google-таблицы через сервис-аккаунт |
| `utils.py` | чистые хелперы (нормализация телефона, сравнение, парсинг) + юнит-тесты |
| `tests/` | юнит-тесты (`python -m unittest discover -s tests`) |
| `Dockerfile`, `docker-compose.yml` | контейнеризация и деплой |
| `htmls/` | сохранённая вёрстка страниц YClients (справочно, для селекторов) |

## Конфигурация (`.env`)

Скопируй `.env.example` в `.env` и заполни. Ключевые переменные:

```ini
BASE_URL=https://yclients.com
SALON_ID=1971030                     # ID филиала (из URL /clients/<ID>/base/)

YCLIENTS_LOGIN=...                    # email/логин YClients для автовхода
YCLIENTS_PASSWORD=...                 # пароль (без 2FA — иначе нужен ручной вход)

GOOGLE_CREDENTIALS=service_account.json
SHEET_URL=https://docs.google.com/spreadsheets/d/XXXX/edit
WORKSHEET=                           # имя листа; пусто = первый лист

HEADLESS=false                       # true на сервере; false для ручного входа/отладки
DRY_RUN=true                         # true = пройти сценарий, но НЕ жать «Отправить»
POLL_INTERVAL=60                     # период опроса таблицы, сек

LOG_LEVEL=INFO                       # уровень логов в консоль (в файл всегда DEBUG)
NAV_RETRIES=3                        # ретраи перехода по URL при таймауте/сети
STEP_RETRIES=2                       # ретраи шага с перезагрузкой страницы
SHEETS_RETRIES=3                     # ретраи вызовов Google Sheets API
```

Защита от работы не в том аккаунте (опционально):
`YCLIENTS_REQUIRED_ACCOUNT_TEXT` / `YCLIENTS_FORBIDDEN_ACCOUNT_TEXT` — текст, который
должен / не должен быть виден в кабинете.

> **Безопасность:** `.env`, `service_account.json`, `client_secret.json`,
> `browser_profile/` — в `.gitignore` и `.dockerignore`. Не коммить их и не запекай в образ.

## Google Sheets — доступ

1. Google Cloud → включить **Google Sheets API** → создать **service account** →
   скачать JSON-ключ → положить как `service_account.json` в корень.
2. Расшарить таблицу на email сервис-аккаунта с правом **Редактор**.
3. Вставить ссылку на таблицу в `SHEET_URL` (ID скрипт вытащит сам).

Проверка подключения:

```bash
python sheets.py        # выведет число строк со статусом «новый»
```

Схема таблицы: `A=№  B=имя  C=телефон  D=текст push  E=статус`. Первая строка — заголовки.

## Запуск локально

```bash
python -m venv .venv && . .venv/Scripts/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
playwright install chromium
cp .env.example .env                                # затем заполнить

python -m unittest discover -s tests                # тесты (должны быть зелёными)
python simple_run.py                                # один прогон по TEST_* из .env
python run.py                                       # боевой воркер (polling таблицы)
```

### Первый вход в YClients

Сессия живёт в `browser_profile/` (persistent Chromium-профиль).

- Если заданы `YCLIENTS_LOGIN`/`YCLIENTS_PASSWORD` и на аккаунте нет 2FA/reCAPTCHA —
  воркер логинится сам, в т.ч. в headless. Ничего делать не нужно.
- Если автовход блокирует reCAPTCHA или включена 2FA — запусти один раз с
  `HEADLESS=false`, войди руками (профиль сохранит сессию), дальше можно `HEADLESS=true`.

## Деплой (Docker)

На сервере (Ubuntu + Docker Compose):

```bash
# 1. положить рядом: .env, service_account.json
# 2. прогреть сессию YClients в browser_profile/ (если нужен ручной вход —
#    сделать первый логин локально и скопировать browser_profile/ на сервер,
#    либо положиться на автовход по логину/паролю)
docker compose build
docker compose up -d
docker compose logs -f          # следить за работой
```

Контейнер:
- `restart: unless-stopped` — поднимается после падения/перезагрузки;
- `browser_profile/`, `service_account.json`, `logs/` — смонтированы томами
  (сессия и логи переживают пересборку/рестарт);
- внутри всегда `HEADLESS=true`, `shm_size: 1gb` (тяжёлая SPA).

## Логи и отладка ошибок

- **`logs/app.log`** — ротируемый файл (10 MB, хранение 14 дней, zip). В файл пишется
  уровень DEBUG, в консоль — `LOG_LEVEL`. Том смонтирован → файл виден на хосте.
- **`logs/artifacts/`** — при каждой ошибке/ретрае сохраняются `<таймштамп>_<тег>.png`
  (полный скриншот) и `.html` (DOM страницы). Это главный инструмент разбора, когда
  меняется вёрстка YClients или не находится элемент.

```bash
docker compose logs -f                 # живой лог
ls logs/artifacts/                     # скриншоты+HTML по ошибкам
```

## Устойчивость (ретраи)

- **Навигация** (`goto_with_retry`): ретраи при таймауте и сетевых ошибках
  (`net::ERR_*`), со сбросом зависшей страницы между попытками.
- **Шаги поиска/создания/открытия формы** (`with_page_retry`): при ненайденном
  элементе страница **перезагружается** и шаг повторяется (`STEP_RETRIES`). Обёрнуты
  только идемпотентные участки — поиск/создание-если-нет и открытие формы push **до**
  необратимой кнопки «Отправить».
- **Отправка push не ретраится** после клика «Отправить» — это защита от дублей пуша.
- **Google Sheets API**: чтение/запись с ретраями (`SHEETS_RETRIES`).
- **Идемпотентность**: статус `в работе` не даёт повторно взять строку при сбое;
  переобработка безопасна (сначала ищем, потом создаём — дубля клиента не будет).

## Известные ограничения

- Селекторы привязаны к вёрстке YClients — при её изменении смотри `logs/artifacts/`
  и правь селекторы в `simple_run.py`.
- 2FA/reCAPTCHA на входе требуют разового ручного логина в persistent-профиль.
- `нет push-канала` детектируется только если UI это показывает; иначе push уходит,
  а доставку решает YClients.
